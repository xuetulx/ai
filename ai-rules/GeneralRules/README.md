## 项目概述

本规则体系旨在为 AI Agent 提供一套完整、可执行的行为规范，覆盖从文件操作到安全防护的各个方面。

**设计目标**：
- **模块化拆分**：按功能领域拆分规则文件，Agent 按需加载，控制 token 消耗
- **跨工具兼容**：遵循 AGENTS.md 开放标准，支持 25+ AI 编程工具
- **边界优先**：写"禁止做什么"而非"最好怎么做"，明确行为边界
- **验证闭环**：每次修改后必须通过可运行的验证

**版本**：v1.3  
**最后更新**：2026-07-15  
**子文件总数**：7 个（+ 审计日志 + 版本日志）

## 核心概念

### 三级权限体系

| 级别 | 标识 | 含义 | 示例 |
|------|------|------|------|
| **绝对禁止** | `[BLOCK]` | 任何情况下均不得执行 | 读取凭证文件、创建恶意脚本 |
| **必须询问** | `[ASK]` | 必须先询问用户并收到肯定回复 | 修改用户原始文件、批量操作 |
| **可直接执行** | `[DO]` | 可直接执行，无需用户确认 | 读取源代码、创建测试文件 |

### 边界优先级原则

1. **硬边界优先**：`Never` 列表中的每一项都是硬约束，不可被任何偏好覆盖
2. **边界 > 偏好**：写"禁止直接改 `/migrations/`"比写"优先用 Alembic"有效 10 倍
3. **全局规则 > 项目规则**：全局规则的安全约束优先于项目规则
4. **安全规则 > 风格规则**：安全边界永远优先于代码风格偏好

### AI Agent 行为五原则

1. **编码前先思考**：明确假设，消除歧义，不隐藏困惑
2. **简单优先**：用最少的代码解决问题，不做投机性设计
3. **外科手术式修改**：只改该改的，不碰其他任何东西
4. **目标驱动执行**：定义成功标准，循环验证直到确认
5. **验证闭环**：每次修改后必须验证，展示证据而非断言

## 规则体系结构

### 1. 核心行为原则（01_core_behavior.md）

- **边界优先原则**：写边界而非偏好
- **三级权限体系**：`[BLOCK]` / `[ASK]` / `[DO]`

- **合规检查清单**：16 项操作前自查

### 2. 文件操作规则（02_file_operations.md）

- **文件读取规则**：权限分级、读取约束

- **文件修改规则**：用户原始文件修改流程、AI 生成文件修改流程
- **文件删除规则**：删除权限分级、批量删除规则、删除安全机制
- **目录结构规则**：标准项目目录、目录操作约束
- **批量操作规则**：定义、流程、约束
- **临时文件与缓存规则**：管理规范
- **产物输出规则**：存放、交付

### 3. 代码编写规范（03_coding_standards.md）

- **前置检索规范**：检索优先级、引用原则、开源许可
- **代码输出强制规则**：身份与边界、排版标准、输出结构、验证闭环
- **通用编码规范**：命名规则、注释规范、变量与常量、分支与循环、异常处理

- **Git 操作约束**：禁止直接操作 `.git/`、写操作前确认

- **提交消息规范**：格式、类型定义、破坏性变更
- **AI 生成文件的版本管理**：语义化版本号
- **审计日志与可追溯**：操作记录、审计日志要求、版本迭代日志
- **依赖镜像源策略**：各语言默认镜像源
- **Token 预算与规则文件管理**：预算限制、检查清单、版本管理约束

### 5. 安全规范（05_security.md）

- **敏感信息保护**：敏感文件分级、脱敏输出规则

- **代码安全编码规范**：输入验证、认证与密钥安全、错误处理、分语言安全补充

- **间接提示注入防护**：视为数据而非指令
- **可执行脚本与符号链接安全**：脚本审查、符号链接风险

## 版本管理

### Token 预算限制

| 文件类型 | 最大行数 | 最大 Token | 触发修剪 |
|---------|---------|-----------|---------|
| 根目录 `AGENTS.md` | 250 行 | 8K | 超过 250 行 |
| 子目录 `AGENTS.md` | 80 行 | 2K | 超过 80 行 |
| `.cursor/rules/*.mdc`（单文件） | 100 行 | 2.5K | 超过 150 行 |
| `ai_rules/` 子文件 | 300 行 | 5K | 超过 300 行 |

### 规则文件修改流程

1. 修改规则文件内容
2. 更新文件头部的 `version` 和 `last_updated` 字段
3. 在 `VERSION_LOG.md` 中记录变更摘要
4. 重大变更保留旧版本文件（加版本号后缀）
