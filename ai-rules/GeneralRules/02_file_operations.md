# 02 文件操作规则

> **适用范围**：创建、读取、修改、删除文件/目录，批量操作，临时文件，产物输出。
>
> **前置依赖**：`01_core_behavior.md`（权限体系 + 边界优先）
>
> **来源**：AgentAI_项目文件操作规则 v2.3 §2~§9；社区 AGENTS.md 六大核心领域对齐

## 1. 文件读取规则

### 1.1 读取权限

| 文件类型 | 权限 | 注意事项 |
|---------|------|---------|
| 源代码文件 | `[DO]` | 不得将敏感代码泄露到外部 |
| 配置文件（不含密钥） | `[DO]` | 敏感值用 `[REDACTED]` 替代后输出 |
| 文档文件 | `[DO]` | 尊重版权和隐私 |
| 依赖目录文件 | `[DO]` | 不修改依赖包内的文件 |
| `.git/` 内部 | `[BLOCK]` | 使用 git 命令间接获取信息 |
| 环境变量文件 (`.env`/`.env.*`) | `[ASK]` | 读取后不得输出明文，敏感值须脱敏 |
| 凭证/密钥/证书 | `[BLOCK]` | 绝对禁止，不可通过 ASK 绕过 |
| 含密码的配置文件 | `[ASK]` | 读取时脱敏输出，密码字段用 `***` 替代 |

### 1.2 读取约束

1. `[DO]` 大文件（> 10MB）优先分页/分段读取
2. `[DO]` 二进制文件只在必要时读取，不输出原始内容
3. `[DO]` 读取时保持只读模式，不修改文件时间戳或属性
4. `[ASK]` 批量读取 ≥ 20 个文件时，先告知用户范围和目的

## 2. 文件创建规则

### 2.1 创建位置规范

| 文件类型 | 推荐目录 | 说明 |
|---------|---------|------|
| 源代码 | `src/`、`lib/`、`utils/`、`modules/` | 按语言和功能分子目录 |
| 测试代码 | `tests/`、`test/`、`__tests__/` | 与源码结构对应 |
| 输出产物 | `output/`（首选）; `dist/`、`build/`、`result/` | AI 任务最终产出 |
| 临时文件 | `tmp/`、`.temp/`、`.cache/` | 任务结束后自动清理 |
| 文档说明 | `docs/`、`doc/`、`notes/` | Markdown 格式优先 |
| 配置文件 | `config/`、`conf/`、根目录（仅少量） | 不与用户配置冲突 |
| 数据文件 | `data/`、`dataset/`、`assets/` | 按数据类型分 |
| 脚本工具 | `scripts/`、`tools/`、`bin/` | 可执行脚本统一管理 |
| AI 指令文件（通用） | `AGENTS.md`（根目录）; 子目录/`AGENTS.md` | 跨工具通用标准（首选） |
| Claude Code 规则 | `.claude/rules/`（模块化）; `CLAUDE.md`（薄适配层） | Claude Code 专用 |
| Cursor 规则（2026） | `.cursor/rules/*.mdc` | Cursor 推荐格式，替代 `.cursorrules` |
| GitHub Copilot 规则 | `.github/copilot-instructions.md` | GitHub Copilot 专用 |

### 2.1.1 .mdc 文件创建规范（v1.1 新增）

> Cursor 2026 年推荐 `.cursor/rules/*.mdc` 格式，已弃用 `.cursorrules`

```yaml
---
description: 规则描述（显示在 Cursor UI 中）
globs: "src/**/*.ts"          # glob 作用域，限制规则生效的文件范围
alwaysApply: false             # true=Always 模式；false=Agent Requested 模式
enabled: true                  # 是否启用
---
# 规则内容（Markdown 格式）
```

**四种激活模式**：
| 模式 | `alwaysApply` | 说明 |
|------|--------------|------|
| **Always** | `true` | 每次对话都加载，用于全局安全/边界规则 |
| **Agent Requested** | `false`（默认） | Agent 根据任务描述自动判断是否需要，推荐用于领域规则 |
| **Manual** | 需在 Cursor 中手动 `@` 引用 | 用于临时规则 |
| **Auto Attached** | 通过 `globs` 自动匹配 | 文件匹配时自动附加 |

> **建议**：安全/边界规则用 Always 模式 + `alwaysApply: true`；代码风格规则用 Agent Requested 模式 + `globs` 精确匹配。

### 2.2 文件命名规范

1. `[DO]` 小写字母 + 下划线/连字符，禁止中文文件名、空格、特殊字符
2. `[DO]` AI 生成文件加前缀标识：`ai_gen_xxx.py`、`ai_output_xxx.md`
3. `[BLOCK]` AI 生成的源代码/重要产物文件**必须**在文件名中包含语义化版本号（如 `radar_bk_stats_v1.2.py`），禁止创建不含版本号的生产文件
4. `[DO]` 临时文件用 `.tmp` 后缀或放入 `tmp/`
5. `[BLOCK]` 禁止创建与用户已有文件同名且会覆盖的文件

### 2.3 受保护文件清单（修改前必须 `[ASK]`）

| 类别 | 文件/目录 |
|------|----------|
| **AI 指令/规则文件** | `CLAUDE.md`、`AGENTS.md`、`AI_RULES_*.md`、`.cursorrules`（已弃用）、`.cursor/rules/*.mdc`、`.cursor/rules/` |
| **项目元文件** | `README.md`、`LICENSE`、`CHANGELOG.md`、`VERSION_LOG.md` |
| **构建/依赖配置** | `package.json`、`Cargo.toml`、`go.mod`、`pyproject.toml`、`requirements.txt` |
| **依赖锁定文件** | `package-lock.json`、`yarn.lock`、`Cargo.lock`、`go.sum`、`pnpm-lock.yaml` |
| **CI/CD 配置** | `.github/workflows/`、`.gitlab-ci.yml`、`Jenkinsfile`、`Dockerfile` |
| **环境/密钥** | `.env`、`.env.*`、`*.pem`、`*.key`、`*.cert`、`config*.yaml` |
| **Git 配置** | `.gitignore`、`.gitattributes`、`.git/config` |
| **编辑器配置** | `.vscode/`、`.idea/`、`.editorconfig` |
| **Claude Code 配置** | `.claude/settings.json`、`.claude/hooks/` |

## 3. 文件修改规则

### 3.1 用户原始文件修改流程

```
[DO]  1. 读取原文件，理解内容
[DO]  2. 创建备份副本：原文件名.bak 或 原文件名_backup_日期
[DO]  3. 在副本或新文件上进行修改
[ASK] 4. 向用户展示修改差异和修改理由，确认是否应用
[DO]  5. 用户确认后覆盖原文件（保留备份或根据用户指示删除）
```

### 3.2 AI 生成文件修改流程

```
[DO]  1. 读取 AI 之前生成的文件，理解内容
[DO]  2. 创建备份副本：原文件名_v{旧版本}.bak 或 原文件名_backup_日期
[DO]  3. 在副本基础上修改，文件名中的版本号递增（v1.0 → v1.1）
[DO]  4. 修改后在回复中说明变更点
[ASK] 5. 修改完成后询问用户是否删除旧版本文件（保留至少一个历史版本）
```

> **v1.4 变更**：AI 生成文件的修改流程从「直接修改」升级为「先备份→递增版本号→修改→询问是否删旧版」，确保所有修改可追溯、可回退。

### 3.3 修改约束

0. `[BLOCK]` **备份前置强制规则**：禁止在未创建 `.bak` 备份文件的情况下对任何现有文件执行 `replace_in_file` 或 `write_to_file`（覆盖模式）。修改任何现有文件前必须完成以下步骤，缺一不可：
   - (a) **创建备份**：复制原文件为 `原文件名_v{当前版本号}.bak` 或 `原文件名_backup_{YYYYMMDD}.bak`
   - (b) **验证备份**：确认备份文件存在且内容与修改前原文件一致（使用 `dir`/`ls` 验证文件存在，使用 `fc`/`diff` 验证内容一致）
   - (c) **执行修改**：方可对原文件执行 `replace_in_file` 或 `write_to_file`
   - (d) **事后告知**：修改完成后在回复中告知用户备份文件路径，询问是否保留或删除备份
1. `[DO]` 保持原有代码风格和缩进格式，不随意重格式化
2. `[DO]` 新增代码遵循 `03_coding_standards.md` 规范
3. `[BLOCK]` 增量修改优先，保留原有逻辑，不做无关重构
4. `[BLOCK]` 保留文件原有的注释和文档
5. `[BLOCK]` 不得删除用户写的注释或 TODO/FIXME 标记
6. `[BLOCK]` 每行变更必须能追溯到用户请求——不做「顺手改进」
7. `[BLOCK]` 发现无关问题提一下但不主动修改
8. `[DO]` 自己的修改导致孤立代码（未使用的 import/变量），自行清理

### 3.4 文件同步规则（v1.3 新增）

> **适用场景**：需要将一个文件（原文件）的变更同步至另一个文件（目标修改文件）时，严格遵循下述处理优先级逻辑。

#### 处理优先级

```
[DO] 步骤 1 — 全文差异比对
  对原文件与待同步修改文件做完整文本差异比对，生成 diff 变更片段（增/删/改）。
  仅标记实际发生变动的段落，未改动文本不纳入变更范围。

[DO] 步骤 2 — 判定改动占比
  基于 diff 结果计算改动内容占比：
    改动占比 = (变更行数 / 目标文件总行数) × 100%

  分支 A：改动占比 < 70%
      → [DO] 局部补丁更新
        仅针对 diff 标记的变动段落，对目标文件做精准局部替换/插入/删除。
        不读写、不输出未改动文本段落。
        目的：大幅减少交互 token 消耗，同时避免 AI 篡改无变更内容。

  分支 B：改动占比 ≥ 70%
      → [DO] 全量覆写（例外放行）
        经判定两份文件改动占比 ≥ 70%，局部补丁无实际收益时，
        允许直接输出完整新版本内容，从头覆写目标文件。
        需在操作说明中明确标注"改动占比 XX%，触发全量覆写例外"。

[BLOCK] 步骤 3 — 约束红线
  无前置差异比对步骤（步骤 1）的，**禁止**直接全量重写目标文件。
  任何情况下不得跳过 diff 生成直接执行全量覆盖。
```

#### 核心原则

| 优先级 | 规则 | 类型 |
|--------|------|------|
| **基础默认** | 先 diff 后局部打补丁，仅更新变动段落，不触碰未改动内容 | `[DO]` |
| **例外放行** | 改动占比 ≥ 70% 时，允许全量覆写目标文件 | `[DO]` |
| **约束红线** | 无前置 diff 比对，不得直接全量重写文件 | `[BLOCK]` |

> **设计意图**：通过强制差异比对的"门槛"机制，最大程度减少不必要的全量文本输出和 token 消耗，
> 同时杜绝 AI 在文件同步场景下擅自篡改未被请求变动的原始内容。

## 4. 文件删除规则

### 4.1 删除权限分级

| 文件类型 | 权限 | 说明 |
|---------|------|------|
| AI 生成的临时文件 | `[DO]` | 任务结束后自动清理 |
| AI 生成的产物文件 | `[DO]` | 建议保留至少一个版本 |
| 输出目录内的文件 | `[DO]` | `output/`、`dist/`、`build/`、`result/` |
| 缓存目录文件 | `[DO]` | `.cache/`、`tmp/` |
| 用户原始文件（1~2 个） | `[ASK]` | 不得擅自删除 |
| 受保护文件 | `[ASK]` | 需用户明确同意 |
| 依赖目录文件 | `[BLOCK]` | 通过包管理器操作 |
| `.git/` 目录 | `[BLOCK]` | 会导致版本历史丢失 |

### 4.2 批量删除规则

- **1~2 个文件**：按上表权限执行
- **3~4 个文件**：`[ASK]` 列出待删除清单（路径、类型、修改时间），确认后执行
- **≥ 5 个文件**：`[BLOCK]` 逐条展示详情，**逐一**获得确认

### 4.3 删除安全机制

1. `[DO]` 删除前确认文件路径和内容
2. `[DO]` 重要文件先移动到 `.trash/` 或 `_deleted/`，不直接永久删除
3. `[BLOCK]` 禁止 `rm -rf` 递归强制删除（`tmp/`、`.cache/`、`.temp/` 除外）
4. `[DO]` git clone 临时副本用完后，删除整个 `tmp/` 目录（非仅子目录）
5. `[DO]` 临时文件操作完成后，用 `Test-Path` 或 `LS` 验证已彻底移除 — **强制步骤**，已纳入 `01_core_behavior.md` 验证闭环步骤 8

## 5. 目录结构规则

### 5.1 标准项目目录

```
项目根目录/
├── AGENTS.md      # 跨工具 Agent 指令文件（首选）
├── CLAUDE.md      # Claude Code 专用指令
├── src/           # 源代码
├── lib/           # 库文件
├── tests/         # 测试代码
├── docs/          # 文档
├── config/        # 配置文件
├── scripts/       # 脚本工具
├── data/          # 数据文件
├── assets/        # 静态资源
├── output/        # 输出产物（AI 可写，首选）
├── dist/          # 构建产物（AI 可写）
├── build/         # 构建产物（AI 可写）
├── result/        # 结果产物（AI 可写）
├── tmp/           # 临时文件（AI 可写）
├── .temp/         # 临时文件（AI 可写）
├── .cache/        # 缓存文件（AI 可写）
├── .claude/       # Claude Code 配置（只读）
├── .cursor/       # Cursor 配置（只读）
├── node_modules/  # 依赖（只读）
├── .venv/         # 虚拟环境（只读）
└── .git/          # Git 目录（禁止直接操作）
```

### 5.2 目录操作约束

1. `[DO]` 优先使用已有目录结构
2. `[ASK]` 新增顶层目录时说明用途并询问
3. `[DO]` 子目录遵循已有命名风格（全小写 + 下划线/连字符）
4. `[BLOCK]` 不得重命名或移动核心目录

## 6. 批量操作规则

### 6.1 批量操作定义

| 操作类型 | 阈值 |
|---------|------|
| 创建文件 | ≥ 10 个 |
| 修改用户原始文件 | ≥ 5 个 |
| 删除文件 | ≥ 3 个 |
| 创建新目录 | ≥ 5 个 |
| 顶层目录结构调整 | ≥ 3 个 |

### 6.2 批量操作流程

```
[DO]  1. 列出所有待操作文件/目录清单
[DO]  2. 说明操作目的和预期影响
[ASK] 3. 询问用户是否确认执行
[DO]  4. 用户确认后分批执行，每批 ≤ 5 个操作
[DO]  5. 执行完成后汇总报告成功/失败情况
```

### 6.3 批量操作约束

1. `[DO]` 操作前先备份或创建 Git 提交点
2. `[DO]` 遇到错误立即停止
3. `[DO]` 操作完成后验证结果正确性

## 7. 临时文件与缓存规则

### 7.1 临时文件管理

1. `[DO]` 统一放入 `tmp/`、`.temp/` 或 `.cache/`
2. `[DO]` 命名加时间戳或任务 ID
3. `[DO]` 任务完成后自动清理 — **强制步骤**，已纳入 `01_core_behavior.md` 验证闭环步骤 8
4. `[DO]` 长时间运行任务定期清理过期文件（> 24 小时）

### 7.2 缓存文件管理

1. `[DO]` 放入 `.cache/`，按模块分子目录
2. `[DO]` 命名包含版本标识
3. `[DO]` 读取前验证有效性
4. `[DO]` 不得将敏感信息存入缓存

## 8. 产物输出规则

### 8.1 产物存放

1. `[DO]` 优先放入 `output/` 目录
2. `[DO]` 亦可放入 `dist/`、`build/`、`result/`
3. `[DO]` 命名清晰，包含任务标识和版本号
4. `[DO]` 多文件产物创建子目录分类存放

### 8.2 产物交付

1. `[DO]` 明确告知文件路径和文件名
2. `[DO]` 重要产物附带简要说明
3. `[DO]` 多产物列清单
4. `[DO]` 确保产物文件完整、可运行 — **强制步骤**，已纳入 `01_core_behavior.md` 验证闭环步骤 9
