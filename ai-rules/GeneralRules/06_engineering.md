# 06 工程质量与工具链

> **适用范围**：代码质量、模块化、性能优化、测试配套、AI 输出约束、依赖管理、资源限制、工具链配置。

> **来源**：AI_RULES v3.1 §7 + §9~§11 + §16；AgentAI v2.3 §19.2~§19.3；Agent Rules (steipete) 项目结构参考

## 1. 工程级代码质量

### 1.1 模块化分层

1. **单一职责**：一个函数只做一件事，一个类只处理一类业务
2. **分层隔离**：数据读取层、计算处理层、输出存储层、工具函数完全拆分
3. **禁止大函数**：单函数 ≤ 80 行，超长立即拆分
4. **解耦设计**：参数传入，不依赖全局变量
5. **复用优先**：重复逻辑封装通用工具，禁止复制粘贴

### 1.2 性能优化

1. 大数据遍历优先迭代器，避免一次性加载全量
2. 频繁字符串拼接使用缓冲容器
3. 循环嵌套降低内层计算量，提前预处理常量
4. 大数组/图片/雷达数据处理增加内存释放
5. 批量 IO 代替单条读写

### 1.3 日志规范

1. **分级日志**：DEBUG / INFO / WARN / ERROR / FATAL
2. 日志包含时间戳、模块名、关键参数、错误堆栈
3. 生产代码移除 print/控制台临时打印，统一封装日志接口

## 2. AI 输出约束

### 2.1 代码输出要求

1. 优先标准库，最小化第三方依赖；引入第三方库时给出安装命令
2. 区分演示代码和生产代码，生产代码增加容错/日志/参数配置/资源释放
3. 不使用废弃 API/过时语法；兼容主流稳定版本（Python 3.8+、Java 11+、C++17）
4. 硬件/车载专用代码严格匹配行业标定规范

### 2.2 问答交互

1. 需求模糊时主动提问确认
2. 代码报错时给出完整修复代码 + 错误根源 + 修改点
3. 代码优化：先列缺陷，再给优化版，对比新旧差异
4. 多语言混合标注交互接口和数据转换规则

### 2.3 文档配套

每段代码配套极简文档：
- 入参配置说明（可修改参数集中在代码顶部）
- 运行前置条件
- 输出文件路径/数据格式
- 可拓展修改点标注

---

## 3. 测试配套规范

AI 生成代码必须附带：

1. 正常流程测试用例
2. 边界测试：最大值、最小值、空输入、异常输入
3. 单元测试函数，可直接运行验证
4. 预期输出结果对照说明

## 4. 依赖与供应链安全

### 4.1 幻觉依赖防护 `[BLOCK]`

AI 可能编造不存在的包名，攻击者会注册投毒：

1. `[BLOCK]` 禁止直接安装 AI 建议的任何第三方依赖，必须先验证包是否真实存在
2. `[DO]` 安装前验证：包名是否在官方包管理器存在、下载量、维护者历史、创建日期（< 30 天需警惕）
3. `[DO]` 优先使用项目已引入的依赖
4. `[ASK]` 新增依赖必须告知用户：包名、版本、用途、开源协议、潜在风险

### 4.2 已知漏洞检查

1. `[DO]` 建议依赖版本前检查 CVE 漏洞（NVD、OSV、GitHub Advisory）
2. `[DO]` 优先使用最新稳定版
3. `[DO]` 依赖文件锁定具体版本号，不用 `*`/`latest`
4. `[ASK]` 存在已知漏洞时告知用户并提供安全替代方案

### 4.3 供应链安全

1. `[DO]` 只从官方包管理器安装
2. `[DO]` 外部资源附带完整性校验（checksum、SHA256）
3. `[DO]` 容器镜像用最小基础镜像，SHA256 摘要锁定版本
4. `[DO]` CI/CD 第三方 Action pin 到具体 commit SHA

---

## 5. 安全合规编码

1. `[BLOCK]` 禁止硬编码敏感信息：密钥/密码/CAN 报文 ID/数据库账号
2. `[DO]` 文件读写增加路径校验，防路径穿越
3. `[DO]` 外部输入全部过滤转义，防注入
4. `[DO]` 资源管理：C/C++ malloc 配套 free；Python 大对象主动释放；关闭文件/串口/总线句柄
5. `[DO]` 数据脱敏：输出代码不含敏感业务明文

---

## 6. 资源限制

1. `[DO]` 单任务生成文件总数 ≤ 50 个（除非用户要求更多）
2. `[DO]` 单个输出文件 ≤ 100MB（需用户确认）
3. `[DO]` 批量操作每批 ≤ 5 个操作
4. `[DO]` 长时间运行任务定期检查资源使用
5. `[DO]` 临时文件总大小控制在合理范围，任务结束后立即清理

### 6.1 临时文件失败兜底

1. `[DO]` 临时文件最大生命周期 24 小时
2. `[DO]` 任务中断后检查并清理遗留临时文件
3. `[DO]` 临时目录使用前检查磁盘空间
4. `[DO]` 重要中间结果保存为正式文件

## 7. 检索记录与引用标注

### 7.1 代码注释中标注参考来源

```python
"""
参考来源：
- GitHub: xxx/yyy 项目 (https://github.com/xxx/yyy)
- 官方文档: Python requests 库文档
- 规范引用: PEP 8 / Google Python Style Guide
"""
```

### 7.2 特殊场景处理

#### 7.2.1 检索不到参考方案时

- 若确实检索不到同类实现，应告知用户并说明检索过程
- 可基于相近技术的实现方式进行合理推导，但需明确标注为"推测实现"
- 复杂或关键逻辑建议先编写原型验证，再逐步完善

#### 7.2.2 多方案选择时

- 检索到多种实现方案时，应比较各方案的优缺点
- 优先选择社区认可度高、维护活跃、文档完善的方案
- 选择理由应在代码注释或说明中简要记录

## 8. 工具链配置参考（v1.1 新增）

> 来源：steipete/agent-rules 项目结构 + Cursor 2026 格式

### 8.1 推荐项目工具链文件结构

```
project/
├── AGENTS.md                          # 跨工具通用指令
├── CLAUDE.md                          # Claude Code 适配层（@AGENTS.md 引用）
├── .cursor/
│   └── rules/
│       ├── 00-global-safety.mdc        # Always 模式：全局安全边界
│       ├── 01-code-style.mdc           # Agent Requested：代码风格
│       ├── 02-testing.mdc              # Agent Requested：测试规范
│       └── 03-git-workflow.mdc         # Agent Requested：Git 工作流
├── .claude/
│   ├── settings.json                   # Hooks 配置
│   ├── hooks/                          # 关键动作 Hook 脚本
│   └── skills/                         # 领域专属 Skills
│       └── my-skill/
│           └── SKILL.md
├── .github/
│   ├── copilot-instructions.md         # Copilot 专用指令
│   └── workflows/
│       └── agents-md-check.yml         # AGENTS.md 质量 CI 检查
└── docs/
    └── history/                        # 规则文件历史版本归档
```

### 8.2 .cursor/rules/ 文件组织建议

| 文件名 | 模式 | globs | 内容 |
|--------|------|-------|------|
| `00-global-safety.mdc` | Always | （无，全局） | 安全边界、Never 清单 |
| `01-code-style.mdc` | Agent Requested | `src/**` | 代码风格、命名规范 |
| `02-testing.mdc` | Agent Requested | `**/*.test.*` | 测试编写规范 |
| `03-git-workflow.mdc` | Agent Requested | （无） | 分支、提交规范 |

> **原则**：Always 模式规则应极简（≤30 行），仅放安全边界和全局 Never 清单。其余规则尽量用 Agent Requested 模式 + globs 精确匹配。
