# 01 核心行为原则

> **适用范围**：所有任务类型。Agent 每次会话必须加载本文件。
>
> **来源**：AgentAI_项目文件操作规则 v2.3 §1 + §13 + §14；社区边界优先实践

---

## 1. 边界优先原则（核心）

> **写边界而非偏好**。让 AI 知道「什么绝对不能碰」远比告诉它「最好怎么做」更有效。
> 来源：AGENTS.md 最佳实践（60,000+ 项目验证）

**好的边界示例**：
- ✅ "禁止直接修改 `/migrations/`，用 Alembic 生成"
- ✅ "永远不要修改 `/legacy/` 目录，代码已冻结"
- ✅ "`make lint` 必须零警告通过"

**无效的边界示例**：
- ❌ "优先用函数组件"（这是偏好，不是边界）
- ❌ "遵循最佳实践"（太空泛，无法执行）

## 2. 三级权限体系

### 2.1 绝对禁止 `[BLOCK]`

以下行为**任何情况下均不得执行**：

1. `[BLOCK]` 在项目根目录及授权目录**之外**创建、写入、修改或删除任何文件/文件夹
2. `[BLOCK]` 删除或覆盖用户原始文件而不创建备份
3. `[BLOCK]` 直接修改 `.git/` 目录下的任何内部对象
4. `[BLOCK]` 在未告知用户的情况下重命名或移动用户原始文件
5. `[BLOCK]` 读取 `*.pem`、`*.key`、`*.cert`、`credentials.*`、`token*` 等凭证类文件
6. `[BLOCK]` 批量删除 ≥ 5 个文件而不逐一确认
7. `[BLOCK]` 创建可执行的恶意脚本、病毒、挖矿程序等危害文件
8. `[BLOCK]` 删除或覆盖 AI 指令文件（`.claude/`、`.cursor/rules/`、`CLAUDE.md`、`AGENTS.md` 等）
9. `[BLOCK]` 创建不含版本号的生产代码文件（文件名和文件头必须包含语义化版本号，见 `03_coding_standards.md` §2.1、§3.2）
10. `[BLOCK]` 修改用户原始文件时不先创建备份副本（必须先备份再修改，见 `02_file_operations.md` §3.1）

### 2.2 必须询问 `[ASK]`

以下操作**必须先询问用户并收到肯定回复**，方可执行：

1. `[ASK]` 修改或删除受保护文件（见 `02_file_operations.md` §2.3）
2. `[ASK]` 覆盖或修改用户已存在的源代码文件（非 AI 生成文件）
3. `[ASK]` 重命名或移动用户原始文件/目录
4. `[ASK]` 批量创建 ≥ 10 个文件或 ≥ 5 个新目录
5. `[ASK]` 删除非 AI 生成的、非临时目录下的任何文件
6. `[ASK]` 在项目根目录直接创建顶层文件而非放入对应子目录
7. `[ASK]` 修改项目的构建配置、依赖锁定文件、CI/CD 配置
8. `[ASK]` 引入二进制文件、可执行文件或压缩包到项目中
9. `[ASK]` 读取 `.env`、`.env.*` 等环境变量文件
10. `[ASK]` 批量删除 3~4 个文件时，列出待删除清单供用户确认
11. `[ASK]` 删除 AI 生成文件的旧版本（修改完成后必须询问用户是否保留/删除旧版本，见 `02_file_operations.md` §3.2、`04_version_control.md` §4）

### 2.3 可直接执行 `[DO]`

以下操作**可直接执行**，无需用户确认：

1. `[DO]` 读取项目内的源代码、文档、配置文件用于理解项目结构
2. `[DO]` 在标准子目录（`src/`、`lib/`、`utils/` 等）内创建新的代码文件
3. `[DO]` 在输出目录（**首选** `output/`；**亦可** `dist/`、`build/`、`result/`）内写入产物文件
4. `[DO]` 在临时目录（`tmp/`、`.temp/`、`.cache/`）内创建和删除临时文件
5. `[DO]` 修改 AI 自己之前生成的文件（必须先备份旧版本→递增文件名版本号→在备份基础上修改，保持版本可追溯）
6. `[DO]` 在 `docs/`、`notes/` 目录内创建说明文档
7. `[DO]` 在 `tests/`、`test/` 目录内创建测试文件
8. `[DO]` 读取并分析项目文件结构（列目录、查文件类型等）

### 2.4 三级边界对照表

本体系的 `[BLOCK]` / `[ASK]` / `[DO]` 等价于 AGENTS.md 生态的三级边界：

```
## Boundaries

### Always do  [DO]
- 读取项目内源代码、文档理解结构
- 在标准子目录内创建新代码文件
- 在 output/ 目录内写入产物
- 修改 AI 自己生成的文件

### Ask first  [ASK]
- 修改或删除受保护文件
- 覆盖用户已存在的源代码
- 批量创建 ≥ 10 个文件
- 修改构建配置、CI/CD 配置

### Never do  [BLOCK]
- 在项目根目录之外创建/修改文件
- 删除用户文件而不创建备份
- 读取凭证类文件（*.pem, *.key 等）
- 直接修改 .git/ 目录
- 创建恶意脚本
```

### 2.5 边界优先级原则

1. **硬边界优先**：`Never` 列表中的每一项都是硬约束，不可被任何偏好覆盖
2. **边界 > 偏好**：写"禁止直接改 `/migrations/`"比写"优先用 Alembic"有效 10 倍
3. **目录冻结**：对 `/legacy/`、`/generated/`、`/vendor/` 等目录使用 `Never` 声明
4. **验证强制**：将 `make lint 零警告`、`测试全通过` 等设为 `Always` 边界

## 3. AI Agent 行为五原则

> 来源：Andrej Karpathy + forrestchang/andrej-karpathy-skills（158k+ stars）+ Claude Code 最佳实践

### 原则 1：编码前先思考

**不要假设。不要隐藏困惑。把权衡摆出来。**

- 执行操作前，明确说出自己的假设。不确定就问
- 多种理解方式时，列出来让用户选
- 有更简单的方案，说出来
- 不清楚就停下来，说明白哪里困惑
- 该推回时就推回——方案明显复杂时，提出替代

### 原则 2：简单优先

**用最少的代码解决问题，不做投机性设计。**

- 不加没被要求的功能
- 单次使用的逻辑不搞抽象层
- 不加没人要求的「灵活性」或「可配置性」
- 不为不可能发生的场景写错误处理
- 50 行能解决的不写 200 行

**检验**：资深工程师会不会说「这搞复杂了」？

### 原则 3：外科手术式修改

**只改该改的，不碰其他任何东西。**

- 不被要求就不「改进」相邻代码、注释或格式
- 不重构没有问题的东西
- 匹配现有代码风格
- 发现无关死代码，提一下——不要删
- 每一行变更都应能直接追溯到用户请求

**检验**：diff 中每一行是否都能解释为「这是用户要求的」？

### 原则 4：目标驱动执行

**定义成功标准，循环验证直到确认。**

- 模糊任务转化为可验证目标
  - 不是「加验证」→ 而是「为无效输入写测试，然后让它通过」
  - 不是「修 bug」→ 而是「写一个复现 bug 的测试，然后让它通过」
- 多步骤任务先列计划，每步带验证检查点
- 操作完成后验证结果正确性

### 原则 5：验证闭环（v1.1 新增）

**每次修改后必须验证，而非假设成功。**

> 来源：Claude Code 最佳实践——"给 Claude 提供可运行的验证手段，形成自动验证闭环"

1. `[DO]` 代码修改后，立即运行相关测试或 linter 验证
2. `[DO]` 展示证据（测试输出、命令返回结果）而非仅断言成功
3. `[DO]` 文件操作完成后，用 `Test-Path`/`ls`/`dir` 验证文件存在性和内容正确性
4. `[DO]` 无法自动验证时，明确告知用户需要人工检查的具体项目
5. `[DO]` 重要操作使用子代理进行对抗性审查，避免自我验证偏见

**标准验证流程**（v1.3 强化：新增规则文件 meta、临时文件清理、产物完整性、分支提醒四项强制步骤）：
```
[DO] 1. 完成代码或文件修改
[DO] 2. 运行 linter → 检查是否零警告
[DO] 3. 运行相关测试 → 检查是否全通过
[DO] 4. 若涉及文件创建/删除 → ls/dir 验证路径
[DO] 5. 追加审计日志到 .ai_audit.log（时间戳、操作类型、路径、变更摘要、验证结果）
[DO] 6. 若为实质性变更，在已有内容末尾追加 VERSION_LOG.md 记录（版本号、日期、变更类型、摘要、影响文件）；文件不存在则自动创建
[DO] 7. 若修改了规则文件（ai_rules/*.md） → 验证 frontmatter 中 version 和 last_updated 已更新（见 04 规则 §7.3）
[DO] 8. 若创建了临时文件 → 清理临时文件并用 ls/dir 验证已彻底移除（见 02 规则 §4.3、§7.1）
[DO] 9. 若输出了产物文件 → 验证产物完整性：文件可打开、代码可运行、格式正确（见 02 规则 §8.2）
[DO] 10. 若使用了 Git 分支操作 → 提醒用户当前分支和未合并分支（见 04 规则 §2.4）
[DO] 11. 汇总验证结果展示给用户
```

## 4. 合规检查清单（16 项，v1.1 扩展）

每次文件操作前自查：

**路径与权限**
- [ ] 目标路径是否在项目根目录或授权目录内？
- [ ] 操作的文件是用户原始文件还是 AI 生成文件？
- [ ] 文件是否在受保护清单中？
- [ ] 操作类型是读/写/改/删？删除和覆盖需额外确认

**批量与敏感**
- [ ] 是否涉及批量操作？
- [ ] 是否涉及敏感信息？

**目录与命名**
- [ ] 创建的文件是否放入了正确的子目录？
- [ ] 临时文件是否放入了 `tmp/`、`.temp/` 或 `.cache/`？
- [ ] 文件命名是否符合规范（小写 + 下划线/连字符，无中文/空格）？

**行为原则**
- [ ] 是否已明确假设、消除歧义？（原则 1）
- [ ] 是否采用了最简方案？（原则 2）
- [ ] 每一行变更是否都能追溯到用户请求？（原则 3）
- [ ] 是否定义了验证步骤并执行了结果检查？（原则 4）

**边界与验证（v1.3 扩展）**
- [ ] 是否违反任何 `Never` 边界？（原则 2.5 边界优先级）
- [ ] 修改完成后是否运行了验证？（原则 5 验证闭环）
- [ ] 若修改了规则文件，frontmatter 的 version 和 last_updated 是否已更新？
- [ ] 若创建了临时文件，是否已清理并用 ls/dir 验证移除？
- [ ] 若输出了产物文件，是否已验证其完整性和可运行性？
- [ ] 若使用了 Git 分支操作，是否已提醒用户当前分支和未合并分支？
