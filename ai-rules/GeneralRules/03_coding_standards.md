# 03 代码编写规范

> **适用范围**：所有代码编写、脚本开发、程序实现类任务。
>
> **前置依赖**：`01_core_behavior.md`（行为五原则 + 验证闭环）
>
> **来源**：AI_RULES_编程参考规范 v3.1 第一部分 §1~§6 + 第二部分 §8 + §12~§13 + §15；Claude Code 验证闭环最佳实践

## 1. 前置检索规范

**编写代码前必须先检索参考，禁止凭空臆造实现方案。**

### 1.1 检索优先级

| 优先级 | 检索源 | 检索内容 |
|--------|--------|---------|
| **P0** | 官方文档 | 语言/框架官方文档、API 参考（PEP 8、Google Style Guides） |
| **P1** | GitHub | 同类项目、参考实现、开源库 |
| **P2** | Stack Overflow | 常见问题、报错排查 |
| **P3** | 技术博客/社区 | 掘金、知乎、CSDN、Medium |
| **P4** | 包管理器仓库 | PyPI、npm、crates.io 使用示例 |

### 1.2 引用原则

- **允许**：参考开源项目架构、借鉴成熟实现、复用官方示例（注明来源）
- **禁止**：编造 API 名称、发明不存在的库、不检索直接写复杂逻辑、抄袭整个项目

### 1.3 开源许可

- 直接复用开源代码必须检查并遵守原项目许可证
- GPL 等传染性许可证代码不可直接用于闭源项目
- MIT、Apache 等宽松许可证需保留版权声明

### 1.4 合规检查清单

每次编写代码前自查：

- [ ] 是否已检索官方文档（PEP 8 / Google Style Guides / 框架官方文档）？
- [ ] 是否已检索 GitHub 上的同类项目/参考实现？
- [ ] 是否已查阅相关库/框架的官方文档？
- [ ] 是否了解该领域的常见最佳实践？
- [ ] 代码中是否存在凭空编造的 API 或函数？
- [ ] 关键逻辑是否有成熟的参考依据？

## 2. 代码输出强制规则

### 2.1 身份与边界

1. 只输出**可运行、无语法错误**代码，禁止伪代码/残缺代码；简化版必须标注
2. 不生成破解、爬虫、木马、越权、隐私窃取类代码
3. 不夹带未说明的第三方付费库/闭源工具；标注开源协议
4. 需求有风险先指出再修正，不能直接按错误需求写
5. 多版本时区分：极简演示版 / 工程完整版 / 高性能生产版

### 2.2 排版标准

1. 代码必须带**完整注释**：文件头、函数、关键分支/算法、复杂变量
2. **新建文件统一 4 空格缩进**，禁止 Tab 与空格混用
3. **修改已有文件保持原缩进风格**
4. 单行 ≤100 字符（Python ≤79），超长自动换行
5. 代码块前后空一行分隔；导入、常量、类、函数、主程序分层隔离

### 2.3 输出结构

```
1. 需求解读（简短）
2. 实现思路/算法说明
3. 完整可运行代码（带注释）
4. 调用示例 & 测试用例
5. 参数说明、返回值说明
6. 依赖安装、运行步骤
7. 边界异常处理说明
8. 优化拓展方案（可选）
```

9. 输出完成后补充：代码可拓展方向、潜在优化点、部署注意事项

### 2.4 验证闭环要求（v1.1 新增）

> 每次代码修改后必须执行验证，不可假设"改完就行"

1. `[DO]` 代码生成后立即运行 linter 检查（如 `ruff`、`eslint`、`golangci-lint`）
2. `[DO]` 运行相关单元测试，确认全部通过
3. `[DO]` 若涉及新功能，运行新增的测试用例
4. `[DO]` 验证结果必须展示给用户（测试输出/命令返回结果），不可仅口头断言
5. `[ASK]` 无法自动验证时，明确告知用户需要人工检查的具体项目和验证方法

**验证检查表**（v1.2 强化）：
```
[ ] Linter 零警告通过
[ ] 相关测试全部通过
[ ] 新增测试覆盖正常和边界场景
[ ] 文件路径/命名符合规范
[ ] 无硬编码敏感信息
[ ] .ai_audit.log 审计日志已追加
[ ] VERSION_LOG.md 版本日志已更新（实质性变更时）
```

## 3. 通用编码规范

### 3.1 命名规则

| 类型 | 规则 | 示例 |
|------|------|------|
| 常量 | 全大写 + 下划线 | `MAX_TIMEOUT`、`PI` |
| 变量/函数 | 小驼峰或蛇形（跟随项目风格） | `userData` / `user_data` |
| 类名 | 大驼峰（PascalCase） | `RadarDataAnalyzer` |
| 私有变量 | 前缀下划线 | `_cache_buf` |
| 布尔变量 | 谓词前缀 | `is_valid`、`has_error` |

- 禁止无意义命名：`a/b/c/tmp`
- 同一项目全程不混用命名风格

### 3.2 注释规范

1. **文件头**：用途、作者、创建时间、输入输出、依赖、修改记录
2. **函数 Docstring**：功能、入参（类型+含义）、返回值、异常、示例
3. **复杂算法/公式**：写明数学逻辑、来源
4. **临时调试代码**标 `# TODO` / `# FIXME`
5. 只解释**为什么这么写**，不重复代码字面意思

### 3.3 变量与常量

1. 魔法数字全部抽离为常量
2. 变量先定义再使用，作用域最小化
3. 只读数据加常量标识；频繁复用数据缓存到局部变量
4. 数组/容器初始化指定长度

### 3.4 分支与循环

1. if/else 必须带大括号
2. 判断常量放左侧：`if 0 == ret_code`
3. 多分支优先 switch/match，嵌套不超过 3 层
4. 循环内不做重复计算、不频繁 IO
5. 提前 return 减少嵌套

### 3.5 异常与边界处理

1. **所有输入必须校验**：空值、越界、类型错误、空文件、NaN
2. 文件/网络/数据库/总线读写全部加异常捕获
3. 给出默认兜底值，程序不可直接崩溃；日志分级（WARN/ERROR/FATAL）
4. 数值运算处理除零、溢出、精度丢失；浮点比较用误差区间

## 4. 分语言特殊规则

> 引用自 Google Style Guides 及各语言官方规范

### 4.1 Python

- 遵循 **PEP 8** + **Google Python Style Guide**，使用类型注解
- 使用 `with` 管理文件/连接
- 字符串引号统一：同一文件中单引号与双引号择一而行（保持一致即可）
- **行长度**：代码 ≤79 字符；docstrings / 注释 ≤72 字符（PEP 8 规定）
- **导入规范**：按标准库 → 第三方库 → 本地应用分组，组间空一行；优先绝对导入，避免相对导入；推荐 `from __future__ import annotations`
- **Trailing commas**：多行序列的末尾元素建议加 trailing comma，便于 diff 阅读
- 推荐校验工具：Ruff、Flake8、Black、mypy、pylint（Google 官方推荐）
- 优先使用标准库，减少第三方依赖

### 4.2 C / C++

- 遵循 **Google C++ Style Guide** + C++ Core Guidelines
- 指针判空，防止内存泄漏，禁用野指针
- 数组越界校验，缓冲区溢出防护
- RAII 原则管理资源，智能指针优先
- 推荐校验工具：clang-format、cpplint、clang-tidy

### 4.3 Java

- 遵循 **Google Java Style Guide** + 阿里巴巴 Java 开发手册
- 判空使用工具类（Objects、Optional）
- 线程安全约束，关闭流资源使用 try-with-resources
- 推荐校验工具：Checkstyle、Alibaba P3C、google-java-format

### 4.4 JavaScript / TypeScript

- 遵循 **Google JS/TS Style Guide** + ESLint + Airbnb 规范
- TypeScript 强类型约束，异步代码统一捕获异常
- DOM 销毁时清理事件监听
- 推荐工具：ESLint、Prettier

### 4.5 Go

- 遵循 **Google Go Style Guide** + Effective Go + gofmt
- 错误显式处理，defer 管理资源释放
- 推荐工具：golangci-lint、go vet

### 4.6 Shell / Bash

- 遵循 **Google Shell Style Guide**
- 变量引用双引号包裹，`$(command)` 代替反引号
- 管道命令 `set -o pipefail`

### 4.7 其他

- **CAPL/CANoe**：报文周期、信号偏移详细注释，总线负载时序约束
- **MATLAB**：矩阵向量化替代循环，坐标轴步进划分
- **Markdown**：遵循 Google Markdown Style Guide，标题不跳级

## 5. 提交消息规范（Conventional Commits）

```
<type>[optional scope]: <description>

[optional body]

[optional footer(s)]
```

| 类型 | 说明 |
|------|------|
| `feat` | 新功能 |
| `fix` | Bug 修复 |
| `docs` | 文档变更 |
| `style` | 代码格式 |
| `refactor` | 代码重构 |
| `perf` | 性能优化 |
| `test` | 测试相关 |
| `chore` | 构建/辅助工具 |
| `ci` | CI/CD 配置 |
| `build` | 构建系统/外部依赖 |
| `revert` | 回退提交 |

破坏性变更在类型/作用域后加 `!` 或 footer 写 `BREAKING CHANGE:`。

## 6. 可复用 Prompt 模板

```
【顶层规则】
1. 输出完整可运行代码，禁止残缺伪代码；需求有风险先指出再修正。
2. 拒绝生成违规代码。
3. 输出结构：需求解读→实现思路→完整代码→调用示例→测试→依赖→异常处理。
4. 编写前必须先检索参考，禁止凭空臆造。

【代码规范】
1. 新建文件 4 空格缩进，单行 ≤100 字符，命名见名知意。
2. 修改已有文件保持原缩进风格。
3. 标准注释，魔法数字定义常量。
4. 所有输入做校验，IO/网络增加异常捕获。
5. 单一职责，函数 ≤80 行，if 嵌套 ≤3 层。
6. 优化内存与性能，资源使用后释放。

【验证闭环】
修改完成后必须：(1) 运行 linter 确认零警告 (2) 运行测试确认全通过 (3) 展示验证结果

【版本控制】
提交消息遵循 Conventional Commits：type(scope): description。
```

## 7. Cursor .mdc 规则编写规范（v1.1 新增）

### 7.1 .mdc 格式模板

为 Cursor 编写代码风格规则时，使用以下模板：

```markdown
---
description: "[语言/框架] 代码风格规范"
globs: "src/**/*.{ts,tsx}"     # 精确匹配目标文件
alwaysApply: false               # Agent Requested 模式
enabled: true
---

# [语言/框架] Coding Standards

## Code Style
- 4 空格缩进，单行 ≤100 字符
- 命名导出，不用默认导出

## Example
```[语言]
// 真实代码示例
function processData(input: string): Result {
    // 标准实现
}
```

## Boundaries
- Never: 使用 `any` 类型（除非加 TODO 注释）
- Never: 直接修改 `generated/` 目录
- Always: 新增函数必须附带单元测试
```

### 7.2 glob 模式最佳实践

| 场景 | glob 示例 |
|------|----------|
| 前端 TypeScript | `"src/**/*.{ts,tsx}"` |
| Python 后端 | `"**/*.py"` |
| 全栈项目（排除依赖） | `"src/**"` |
| 仅测试文件 | `"**/*.test.*"` |
| 配置文件 | `"*.{json,yaml,toml}"` |
| 文档 | `"docs/**"` |

> **原则**：globs 越精确，规则越不会干扰无关文件。Always 模式规则应谨慎使用，仅用于安全/边界类约束。
